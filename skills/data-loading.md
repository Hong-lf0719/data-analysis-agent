# 数据加载

## Overview

加载不是打开文件——是诊断 + 确认 + 选择策略。
**核心原则：损坏的数据加载 = 后续所有分析都是错的。**

<HARD-GATE>
禁止以下行为：
- 禁止在用户确认目标文件之前加载数据
- 禁止在编码失败后自动尝试多种编码——只能报告，让用户选
- 禁止使用 encoding_errors='replace' 或 'ignore' 而不检测数据破坏
- 禁止在多个文件存在时替用户选择
</HARD-GATE>

## Checklist

按顺序完成：

### 1. 列出可用文件

- 如果用户给了文件路径 → 直接进入第 2 步
- 如果用户给了目录/压缩包 → 列出所有文件，每条格式：
  `"[文件名] — [大小] — [类型]"`
  让用户选：编号 or 文件名 or "全部"

### 2. 预览数据

- 先检测文件编码（用 chardet 或读取原始字节判断）
- 读取前 5 行 + 表头
- 输出预览报告：
  ```
  文件名：xxx
  大小：X MB
  检测编码：UTF-8 / GBK / 无法确定
  列数：N
  前 5 行：[显示]
  ```
- 问："这是你要加载的数据吗？"
  不是 → 让用户提供正确路径，回到第 1 步

### 3. 大小检查

| 文件大小 | 操作 |
|---------|------|
| < 10 MB | 直接加载 |
| 10-100 MB | 提醒："文件 X MB，加载约需 Y 秒，继续？" |
| > 100 MB | 提醒 + 建议："建议先抽样加载前 N 行预览，再决定是否全量加载" |

用户确认后进入下一步。

### 4. 编码处理

如果编码检测成功 → 用检测到的编码加载。

如果编码失败：

```
① 报告用户：
   "检测到编码问题：文件声称 [X] 编码，但无法正常解码。
   样本字节：[显示原始十六进制]"

② 列出选项让用户选：
   A) 尝试 utf-8-sig
   B) 尝试 gbk / gb18030
   C) 尝试 latin-1（可能包含乱码）
   D) 用户手动指定编码

③ 等待用户选择——不自动尝试
```

### 5. 容错模式检测

如果用户选择了容错编码（如 latin-1 或 errors='replace'）：

必须检测数据是否被破坏：

```
检测标准（任一条触发 → 报告，不继续）：
  • 出现 � 替换字符超过 5 处
  • 列数与预期不符（如应该 11 列变成了 76 列）
  • 行数异常（如预期 1000 行变成了 6000+ 行）
  • 列名出现乱码或空名
```

检测通过 → 继续。检测不通过 → 回到第 4 步，换编码。

### 6. 加载并报告

- 加载数据
- 输出最终加载报告：
  ```
  加载成功：X 行 × Y 列
  编码：[最终使用的编码]
  列名：[前 5 个列名]
  ```
- 保留原始文件不作任何修改
- 报告输出后立即停止，等待用户指示

<HARD-GATE>
data-loading 的边界：

加载阶段允许的操作：
  - 选择编码 / 替换损坏的解码字节（必须报告给用户）
  - 检测并报告列结构、缺失情况、损坏率

加载阶段禁止的操作（留给 data-cleaning）：
  - 删除行/列
  - 填充缺失值
  - 重命名字段
  - 修改数据值（字符修复、类型转换）

加载完成后，下一步动作只能是报告结果 + 等待用户指示。
</HARD-GATE>

## 常见错误

| 错误 | 正确做法 |
|------|---------|
| 不预览直接加载 | 先输出前 5 行让用户确认 |
| 编码失败后自动试 6 种编码 | 报告用户，列出选项，等用户选 |
| 用 errors='replace' 不检查破坏 | 检测替换字符数量和列结构 |
| 大文件直接全量加载 | 检查大小，>100MB 建议抽样 |
| 多文件时替用户选 | 列出所有文件让用户选 |

## Red Flags

| 想法 | 正确做法 |
|------|---------|
| "试一下 utf-8，不行再试 gbk" | 报告用户，不自动迭代 |
| "replace 模式总能读进来，先读再说" | 读进来可能是坏的——先检测 |
| "文件不大，直接加载" | 超过阈值就提醒 |
| "就一个 CSV，直接读" | 先预览 5 行确认 |
